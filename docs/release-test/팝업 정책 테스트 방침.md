제공해주신 `emergency_policy`, `notice_policy`, `update_policy` 스키마를 바탕으로, 팝업들의 **우선순위, 정책 연동, 예외 처리**를 검증하기 위한 **단위 테스트 방침**을 코딩 에이전트에게 요청할 내용으로 정리해 드립니다.

핵심은 \*\*팝업 관리자(PopupManager)\*\*를 만들고, 이 관리자가 세 가지 정책을 어떤 순서와 조건으로 처리하는지 검증하는 것입니다.

## 🎯 코딩 에이전트 지시 사항 (팝업 정책 테스트)

### 1\. 테스트용 Kotlin Data Class 정의

에이전트가 사용할 Supabase 정책 데이터를 표현하는 Kotlin Data Class를 스키마에 맞춰 정의합니다.

```kotlin
// 1. EmergencyPolicy 스키마 기반
data class EmergencyPolicyConfig(
    val app_id: String,
    val is_active: Boolean,
    val content: String,
    val redirect_url: String,
    val button_text: String,
    val is_dismissible: Boolean // 팝업 닫기 가능 여부
)

// 2. NoticePolicy 스키마 기반
data class NoticePolicyConfig(
    val app_id: String,
    val is_active: Boolean,
    val title: String?,
    val content: String,
    val notice_version: Int // 사용자가 본 버전을 로컬에 저장하여 재노출 방지
)

// 3. UpdatePolicy 스키마 기반
data class UpdatePolicyConfig(
    val app_id: String,
    val is_active: Boolean,
    val target_version_code: Int, // 현재 앱 버전보다 높으면 노출
    val is_force_update: Boolean, // 강제 업데이트 여부
    val release_notes: String?,
    val download_url: String,
    val reshow_interval_hours: Int, // '나중에' 클릭 시 재노출 대기 시간
    val max_later_count: Int // '나중에' 최대 클릭 횟수
)
```

### 2\. Mocking 및 환경 구축 요구사항

테스트 시 **외부 네트워크/DB** 및 **시스템 시간**에 의존하지 않도록 다음 Mock 객체들을 구현합니다.

| 영역 | Mock 객체/클래스 | 목적 |
| :--- | :--- | :--- |
| **정책 데이터** | `MockPolicyRepository` | `EmergencyPolicyConfig`, `NoticePolicyConfig`, `UpdatePolicyConfig` 데이터를 **임의로 설정**하여 반환할 수 있도록 구현. (예: `setMockEmergencyPolicy(config: EmergencyPolicyConfig)`) |
| **로컬 저장소** | `MockSharedPreferences` | 사용자가 팝업을 **닫은 횟수** (`max_later_count` 카운터), **마지막 재노출 시간**, **본 공지 버전**을 저장하고 읽는 동작을 가로채서 제어할 수 있도록 구현. |
| **시스템 정보** | `MockSystemInfo` | 테스트용 현재 앱 버전 (`currentVersionCode`)을 임의로 설정할 수 있도록 구현. |
| **시간 관리** | `MockTimeProvider` | 현재 시간을 설정하여 `reshow_interval_hours` 등의 시간 간격 로직을 검증할 수 있도록 구현. |

### 3\. PopupManager (중앙 관리자) 단위 테스트 요청

세 가지 팝업의 **우선순위**와 **개별 정책**이 정확히 작동하는지 검증하는 테스트를 작성합니다. 테스트는 중앙 제어 로직인 `PopupManager.getPopupToShow()` (가칭) 함수를 검증해야 합니다.

#### A. 우선순위 및 충돌 테스트 (Priority & Conflict)

| 테스트 케이스 | 검증 목표 |
| :--- | :--- |
| **Priority\_Emergency\_Override** | **Emergency**와 **Notice** 정책이 동시에 활성화되었을 때, `PopupManager`가 **반드시 Emergency 팝업**을 반환하는지 검증. |
| **Priority\_ForceUpdate\_Override** | Emergency가 비활성화된 상태에서, \*\*강제 업데이트(`is_force_update = true`)\*\*와 Notice 정책이 활성화되었을 때, **Update 팝업**이 반환되는지 검증. |
| **Policy\_Inactive\_Check** | 모든 정책의 `is_active`를 `false`로 설정했을 때, `PopupManager`가 팝업을 **반환하지 않는지**(`null` 반환) 검증. |
| **Policy\_Data\_Failure** | `MockPolicyRepository`가 데이터를 가져오는 데 \*\*실패(Exception)\*\*했을 때, 앱이 충돌하지 않고 **모든 팝업을 노출하지 않는지** 안전하게 처리하는지 검증. |

#### B. 개별 정책 논리 테스트 (Update & Notice)

| 테스트 케이스 | 검증 목표 |
| :--- | :--- |
| **Update\_Version\_Check** | `currentVersionCode`가 `target_version_code`보다 **크거나 같을 때**, Update 팝업이 **노출되지 않는지** 검증. |
| **Update\_MaxLater\_Check** | 사용자가 **`max_later_count`만큼 '나중에'를 클릭**한 후, 다시 앱을 켰을 때 Update 팝업이 **강제로 노출되는지** 검증. |
| **Update\_Reshow\_Interval** | 사용자가 '나중에' 클릭 시, 로컬에 저장된 시간으로부터 `reshow_interval_hours` (또는 minutes/seconds)가 지나지 않았다면 Update 팝업이 **다시 뜨지 않는지** 검증. (`MockTimeProvider` 활용) |
| **Notice\_Version\_Check** | `notice_version` 5를 로컬 저장소에 기록한 후, Supabase에서 `notice_version` 4의 정책을 가져왔을 때, 해당 Notice 팝업이 **노출되지 않는지** 검증. (이미 본 버전은 스킵) |

#### C. Emergency 정책 테스트

| 테스트 케이스 | 검증 목표 |
| :--- | :--- |
| **Emergency\_Dismissible** | `is_dismissible`이 `false`로 설정되었을 때, 팝업이 뜬 후 사용자가 **뒤로 가기 버튼**을 눌러도 **닫히지 않는지** 확인. |
| **Emergency\_Dismissible\_True** | `is_dismissible`이 `true`로 설정되었을 때, 팝업이 **닫히는지** 확인. |
| **Emergency\_Content\_Check** | `content`, `button_text`, `redirect_url` 필드가 DB에서 가져온 값과 팝업 UI 모델에 **정확히 매핑**되어 전달되는지 검증. |
