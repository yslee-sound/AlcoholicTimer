제공해주신 `emergency_policy`, `notice_policy`, `update_policy` 스키마를 바탕으로, 팝업들의 **우선순위, 정책 연동, 예외 처리**를 검증하기 위한 **단위 테스트 방침**을 코딩 에이전트에게 요청할 내용으로 정리해 드립니다.

핵심은 \*\*팝업 관리자(PopupManager)\*\*를 만들고, 이 관리자가 세 가지 정책을 어떤 순서와 조건으로 처리하는지 검증하는 것입니다.

## 🎯 코딩 에이전트 지시 사항 (팝업 정책 테스트)

### 1\. 테스트용 Kotlin Data Class 정의

에이전트가 사용할 Supabase 정책 데이터를 표현하는 Kotlin Data Class를 스키마에 맞춰 정의합니다.

```kotlin
// 1. EmergencyPolicy 스키마 기반
data class EmergencyPolicyConfig(
    val app_id: String,
    val is_active: Boolean,
    val content: String,
    val redirect_url: String,
    val button_text: String,
    val is_dismissible: Boolean // 팝업 닫기 가능 여부
)

// 2. NoticePolicy 스키마 기반
data class NoticePolicyConfig(
    val app_id: String,
    val is_active: Boolean,
    val title: String?,
    val content: String,
    val notice_version: Int // 사용자가 본 버전을 로컬에 저장하여 재노출 방지
)

// 3. UpdatePolicy 스키마 기반
data class UpdatePolicyConfig(
    val app_id: String,
    val is_active: Boolean,
    val target_version_code: Int, // 현재 앱 버전보다 높으면 노출
    val is_force_update: Boolean, // 강제 업데이트 여부
    val release_notes: String?,
    val download_url: String,
    val reshow_interval_hours: Int, // '나중에' 클릭 시 재노출 대기 시간
    val max_later_count: Int // '나중에' 최대 클릭 횟수
)
```

### 2\. Mocking 및 환경 구축 요구사항

테스트 시 **외부 네트워크/DB** 및 **시스템 시간**에 의존하지 않도록 다음 Mock 객체들을 구현합니다.

| 영역 | Mock 객체/클래스 | 목적 |
| :--- | :--- | :--- |
| **정책 데이터** | `MockPolicyRepository` | `EmergencyPolicyConfig`, `NoticePolicyConfig`, `UpdatePolicyConfig` 데이터를 **임의로 설정**하여 반환할 수 있도록 구현. (예: `setMockEmergencyPolicy(config: EmergencyPolicyConfig)`) |
| **로컬 저장소** | `MockSharedPreferences` | 사용자가 팝업을 **닫은 횟수** (`max_later_count` 카운터), **마지막 재노출 시간**, **본 공지 버전**을 저장하고 읽는 동작을 가로채서 제어할 수 있도록 구현. |
| **시스템 정보** | `MockSystemInfo` | 테스트용 현재 앱 버전 (`currentVersionCode`)을 임의로 설정할 수 있도록 구현. |
| **시간 관리** | `MockTimeProvider` | 현재 시간을 설정하여 `reshow_interval_hours` 등의 시간 간격 로직을 검증할 수 있도록 구현. |

<br>

### 3\. PopupManager (중앙 관리자) 단위 테스트 요청

세 가지 팝업의 **우선순위**와 **개별 정책**이 정확히 작동하는지 검증하는 테스트를 작성합니다. 테스트는 중앙 제어 로직인 `PopupManager.getPopupToShow()` (가칭) 함수를 검증해야 합니다.

#### A. 우선순위 및 충돌 테스트 (Priority & Conflict)

| 테스트 케이스 | 검증 목표 |
| :--- | :--- |
| **Priority\_Emergency\_Override** | **Emergency**와 **Notice** 정책이 동시에 활성화되었을 때, `PopupManager`가 **반드시 Emergency 팝업**을 반환하는지 검증. |
| **Priority\_ForceUpdate\_Override** | Emergency가 비활성화된 상태에서, \*\*강제 업데이트(`is_force_update = true`)\*\*와 Notice 정책이 활성화되었을 때, **Update 팝업**이 반환되는지 검증. |
| **Policy\_Inactive\_Check** | 모든 정책의 `is_active`를 `false`로 설정했을 때, `PopupManager`가 팝업을 **반환하지 않는지**(`null` 반환) 검증. |
| **Policy\_Data\_Failure** | `MockPolicyRepository`가 데이터를 가져오는 데 \*\*실패(Exception)\*\*했을 때, 앱이 충돌하지 않고 **모든 팝업을 노출하지 않는지** 안전하게 처리하는지 검증. |

#### B. 개별 정책 논리 테스트 (Update & Notice)

| 테스트 케이스 | 검증 목표 |
| :--- | :--- |
| **Update\_Version\_Check** | `currentVersionCode`가 `target_version_code`보다 **크거나 같을 때**, Update 팝업이 **노출되지 않는지** 검증. |
| **Update\_MaxLater\_Check** | 사용자가 **`max_later_count`만큼 '나중에'를 클릭**한 후, 다시 앱을 켰을 때 Update 팝업이 **강제로 노출되는지** 검증. |
| **Update\_Reshow\_Interval** | 사용자가 '나중에' 클릭 시, 로컬에 저장된 시간으로부터 `reshow_interval_hours` (또는 minutes/seconds)가 지나지 않았다면 Update 팝업이 **다시 뜨지 않는지** 검증. (`MockTimeProvider` 활용) |
| **Notice\_Version\_Check** | `notice_version` 5를 로컬 저장소에 기록한 후, Supabase에서 `notice_version` 4의 정책을 가져왔을 때, 해당 Notice 팝업이 **노출되지 않는지** 검증. (이미 본 버전은 스킵) |

#### C. Emergency 정책 테스트

| 테스트 케이스 | 검증 목표 |
| :--- | :--- |
| **Emergency\_Dismissible** | `is_dismissible`이 `false`로 설정되었을 때, 팝업이 뜬 후 사용자가 **뒤로 가기 버튼**을 눌러도 **닫히지 않는지** 확인. |
| **Emergency\_Dismissible\_True** | `is_dismissible`이 `true`로 설정되었을 때, 팝업이 **닫히는지** 확인. |
| **Emergency\_Content\_Check** | `content`, `button_text`, `redirect_url` 필드가 DB에서 가져온 값과 팝업 UI 모델에 **정확히 매핑**되어 전달되는지 검증. |

## P1 ~ P4 페이즈별 상세 검증 계획
아래는 팝업 정책(긴급/공지/업데이트)에 대해 단계별로 진행할 검증(자동/수동)을 정리한 것입니다. 각 페이즈별 목적, 준비물, 구체적인 테스트 방법, 통과 조건(정의) 및 산출물을 명시합니다.

### P1 — 단위(Unit)테스트 (자동화, 로컬)
- 목적
  - 팝업 결정 로직(PopupManager)이 스키마 기반 입력에 대해 우선순위·예외 처리를 정확히 수행하는지 빠르게 검증.
- 준비물
  - 이미 추가된 단위 테스트 파일들 (app/src/test/.../PopupManagerTest.kt, PopupManager.kt, PopupMocks.kt, TestModels.kt).
  - JDK, Gradle wrapper.
- 테스트 방법
  1. 로컬에서 Gradle 단위테스트 실행: ./gradlew test (Windows에서는 .\gradlew.bat test).
  2. 테스트 스위트는 다음 시나리오를 포함: Emergency 우선, 강제 업데이트 우선, 비활성 정책 무시, 데이터 실패 처리, 업데이트 재노출 로직(reshow_interval / max_later_count), 공지 버전 체크, 긴급 공지 필드 매핑 등.
- 테스트 통과 의미
  - 모든 단위 테스트가 성공하면 PopupManager의 순수 로직(입력->결정)이 문서화된 규칙대로 동작함을 보장한다.
  - 통과 산출물: Gradle 테스트 리포트(HTML) 및 콘솔 성공 로그.

### P2 — 통합(Integration) 테스트 (네트워크 모의 혹은 스테이징)
- 목적
  - Supabase에서 받아오는 JSON 형식이 실제 리포지토리(EmergencyPolicyRepository, UpdatePolicyRepository, NoticePolicyRepository)에서 올바르게 파싱되어 PopupManager로 전달되는지 검증.
- 준비물
  - Supabase REST 엔드포인트(테스트용) 또는 로컬/CI에서 동작하는 HTTP 모킹(예: MockWebServer) 환경.
  - BuildConfig에 SUPABASE_URL / SUPABASE_KEY가 테스트 환경으로 설정되어 있거나, 레포지토리에서 엔드포인트를 주입할 수 있는 hook.
- 테스트 방법
  1. MockWebServer 또는 테스트 스테이징 Supabase에 테스트 레코드(긴급/공지/업데이트)를 업로드.
  2. 앱의 Repository 계층(EmergencyPolicyRepository 등)을 실제 네트워크 호출 모드로 실행하거나, repo 레이어에 네트워크 엔드포인트를 주입하여 동작시킨다.
  3. PopupPolicyManager.decidePopup(…)를 호출하고, 리턴된 PopupDecision이 예상과 일치하는지 확인.
- 테스트 통과 의미
  - 네트워크 응답 파싱 및 app_id 매칭 로직이 실제 Supabase 스키마와 호환됨을 확인한다.
  - 통과 산출물: 통합 테스트 로그, 모의 서버에 사용된 JSON 샘플, 테스트 결과 리포트.

### P3 — UI/Manual 또는 계측(Instrumentation) 테스트
- 목적
  - 실제 다이얼로그(EmergencyRedirectDialog, AnnouncementDialog, OptionalUpdateDialog)가 정책 데이터에 따라 올바르게 렌더링되고 동작하는지 검증(버튼 동작, 링크 열기, 닫기 동작 등).
- 준비물
  - 실제 디바이스 또는 에뮬레이터(Android Instrumentation tests), Espresso/Compose 테스트 환경 설정.
  - 테스트용 Supabase/stub 또는 앱에 테스트 플래그로 다이얼로그를 강제 표시할 수 있는 디버그 훅.
- 테스트 방법 (자동)
  1. Instrumentation 테스트에서 Repository를 테스트용 Mock으로 주입하거나, 테스트 데이터를 유효한 Supabase JSON으로 구성한다.
  2. 앱을 실행하고, 스플래시 이후 Popup 로직이 작동하도록 시뮬레이션한다.
  3. ComposeTestRule / Espresso로 다이얼로그가 보이는지, 제목·본문·버튼 텍스트·링크 URL이 예상값과 일치하는지 검증.
- 테스트 방법 (수동)
  1. 스테이징 서버에 긴급/공지/업데이트 정책을 올리고 앱을 실행하여 사용자 플로우에서 다이얼로그가 노출되는지 확인.
  2. 긴급 공지의 redirect_url 클릭 -> 외부 브라우저/PlayStore 이동 확인, is_dismissible=false 시 뒤로/아웃사이드 클릭으로 닫히지 않는지 확인.
- 테스트 통과 의미
  - 사용자에게 보이는 UI가 서버 정책과 일치하고, 버튼/링크 행동이 정상임을 보장한다.
  - 통과 산출물: 스크린샷, 테스트 비디오(선택), Espresso/Compose 테스트 리포트.

### P4 — 릴리즈/검증(Policy Validation) 단계
- 목적
  - 릴리즈 빌드 관련 정책 검증(광고 숨김 로직, 릴리즈 광고 ID 설정 등) 및 전체 통합 상태 확인.
- 준비물
  - 릴리즈 빌드 설정(Gradle), 실제 릴리즈 광고 유닛 ID가 build.gradle.kts에 설정되어 있는지 확인.
  - verifyReleaseAdConfig 같은 프로젝트 내 스크립트가 요구하는 체크리스트 점검.
- 테스트 방법
  1. 릴리즈(AAB/APK) 빌드를 수행하거나 릴리즈 검증 스크립트(.\gradlew assembleRelease 또는 verifyReleaseAdConfig)를 실행.
  2. 스크립트가 요구하는 코드 패턴(예: BuildConfig.DEBUG 체크)을 만족하는지 확인하고, 광고 ID가 릴리즈 설정에 포함되어 있는지 검증.
- 테스트 통과 의미
  - 릴리즈 빌드 시 광고 정책 위반 위험(디버그용 광고 숨김 로직이 릴리즈에 남아 있는 등)을 자동으로 차단한다.
  - 통과 산출물: 빌드 로그, verifyReleaseAdConfig 출력 결과.

## 테스트 실행 및 결과 해석 가이드
- 기본 실행 명령
  - 단위 테스트: (Windows) .\\gradlew.bat test
  - 특정 테스트만 실행: .\\gradlew.bat :app:test --tests "kr.sweetapps.alcoholictimer.popup.PopupManagerTest.*"
  - 릴리즈 검증: .\\gradlew.bat verifyReleaseAdConfig
- 결과 해석
  - 단위 테스트 성공: PopupManager의 규칙 기반 결정 로직이 문서와 일치.
  - 통합 테스트 성공: Repository의 파싱/매칭 로직이 서버 스키마와 호환.
  - UI(Instrumentation) 테스트 성공: 실제 사용자에게 노출되는 다이얼로그의 텍스트/버튼/링크/닫기 동작이 예상대로 작동.
  - 릴리즈 검증 성공: 릴리즈에 디버그용 광고 숨김 코드/잘못된 광고 ID가 남아있지 않음.

## 테스트 통과가 의미하는 것(요약)
1. 안정성: 네트워크 오류나 예외 상황에서도 팝업 로직이 앱을 크래시시키지 않고 안전하게 동작.
2. 일관성: 서버에서 전달한 정책 필드가 클라이언트의 의사결정 및 UI에 정확히 반영됨.
3. 규정 준수: 릴리즈 빌드 규칙(광고/숨김 로직, 광고 유닛 설정 등)을 충족하여 정책 위반 위험을 줄임.
