## 📝 광고 정책 단위 테스트 (AdsCombinedTest) **보완 요청 명세서**

### 1. 문서 개요 및 목적

본 문서는 현재 운영 중인 `ad_policy` 단위 테스트 파일 (`AdsCombinedTest.kt`)에 **누락된 핵심 비즈니스 로직** (특히 Supabase 정책 데이터의 **유효성 및 초기 필터링**) 검증 항목을 추가하도록 요청하는 명세서입니다.

* **목표:** 광고 관련 로직이 **정책 데이터의 유효하지 않은 상태**에서도 안전하게 작동하고, **가장 상위의 정책 규칙**(`is_active`, 타입별 활성화 플래그)을 정확히 따르는지 보장합니다.

***

### 2. 테스트 환경 및 대상 (기존 방침 유지)

* **테스트 위치:** `app/src/test/java/kr/sweetapps/alcoholictimer/ads/AdsCombinedTest.kt`
* **테스트 대상 핵심 모듈:** `AdManager`, `AdController`, `AdPolicyRepository`
* **실행 방법:** `./gradlew :app:testDebugUnitTest` (로컬 JVM 단위 테스트)

***

### 3. 보완 및 추가가 필요한 핵심 검증 항목

기존 방침에 명시된 **시간창(hour/day) 초기화, 쿨다운, 카운트 상한 처리** 외에, **`ad_policy` 스키마 데이터의 해석**과 관련된 다음 항목들을 **`AdManager` 및 `AdPolicyRepository` 로직 검증**에 추가해야 합니다.

#### 3.1. 🛡️ 정책 데이터 유효성 및 최상위 필터링 검증 (Mock Data 필수)

| No. | 검증 시나리오 | 대상 로직/함수 | 기대하는 결과 (Pass 조건) |
| :--- | :--- | :--- | :--- |
| **P1** | **정책 전체 비활성화 테스트** | `AdManager.canShowAd()` | Mock Data에서 `is_active=false`인 정책이 주입될 경우, 모든 광고 타입(`APP_OPEN`, `INTERSTITIAL`, `BANNER`)에 대해 **False**를 반환해야 함. (카운트/쿨다운 로직을 실행하지 않아야 함) |
| **P2** | **타입별 활성화 플래그 검사** | `AdManager.canShowAd()` | 정책은 활성화되어 있으나, 특정 타입 플래그(예: `ad_banner_enabled`)가 **False**인 경우, 해당 타입의 광고만 **False**를 반환해야 함. |
| **P6** | **정책 로드 실패 시 안전 대처** | `AdPolicyRepository` 및 `AdManager` 초기화 | Supabase Mock 호출 시, **데이터가 빈 목록**이거나 **파싱 오류**가 발생할 경우, 앱이 **오류 없이** 안전한 **기본 정책 (Default Policy, 예: 모든 광고 비활성화)**으로 초기화되어야 함. |

### 4. 개발팀 유의 사항 및 테스트 요구 사항

1.  **Mock Data 준비:** 위 시나리오(P1, P2, P7 등) 검증을 위해 **`ad_policy` 스키마**를 반영한 **다양한 상태의 Mock Policy Data**가 `AdsCombinedTest.kt` 내부 또는 테스트 리소스로 준비되어야 합니다.
2.  **시간 Mocking 활용:** P1, P2 테스트 시, 기존의 시간 관리 로직(쿨다운, 카운트)이 **False 조건(정책 비활성)**에 의해 실행되지 않는지 확인하여 **테스트 독립성**을 유지해야 합니다.
3.  **코드 리뷰:** 추가된 정책 검증 로직은 **가장 중요한 비즈니스 규칙**을 검증하므로, PR(Pull Request) 시 해당 테스트 코드를 면밀히 검토해 주시기 바랍니다.
